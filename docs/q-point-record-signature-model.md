## 1. Purpose

This document defines a signature model for Q-Point records.

The purpose of the Q-Point Record Signature Model is to make Q-Point records more verifiable, traceable, and reviewable.

Q-Point Protocol v0.2 defines how question-based intellectual contribution may be evaluated through non-monetary value traces, Q-Point components, Catalyst C-vector, and scoring signal modes.

This signature model adds a verification layer.

It answers the following questions:

```text
Who or what generated this Q-Point record?
Which Epicenter ID is it bound to?
Which source content or trace does it refer to?
Which schema version was used?
Which scoring model version was used?
When was the record created?
Has the record been changed after signing?
Was it reviewed, countersigned, or disputed?
```

In short:

```text
Q-Point v0.2:
  Records value traces.

Q-Point signature model:
  Makes value trace records verifiable.
```

---

## 2. Non-Monetary Status

A signed Q-Point record is still non-monetary.

A signature does not convert Q-Point into money.

A signature does not create a legal claim.

A signature does not create ownership.

A signature does not create automatic royalty rights.

A signature does not guarantee future income.

A signature only improves the integrity and verifiability of the record.

Recommended notice:

```text
A signed Q-Point record is a signed non-monetary value trace. The signature verifies record integrity, source binding, or reviewer attestation. It does not represent currency, securities, legal ownership, debt, automatic royalty rights, or a claim to payment.
```

---

## 3. Why Signatures Are Needed

Q-Point records may become reference data for:

* attribution review
* gratitude systems
* licensing review
* value circulation diagrams
* Royalty OS allocation reference
* AI-mediated contribution dashboards
* cross-AI contribution comparison
* future research or citation workflows

If Q-Point records are used as reference data, then record integrity becomes important.

Without signatures, it may be difficult to know:

* whether a record was modified
* whether a score was changed
* whether a source was replaced
* whether the Epicenter ID is authentic
* whether the schema version is correct
* whether a reviewer actually reviewed the record
* whether a system-generated record came from the claimed system

The signature model addresses these risks.

---

## 4. Core Design Principle

The signature model is based on the following principle:

```text
Sign the record.
Bind the source.
Bind the schema.
Bind the scoring model.
Preserve review context.
```

A Q-Point signature should not only sign a score.

It should bind together:

* Q-Point record content
* Epicenter ID
* source hash
* trace reference
* schema version
* scoring model version
* timestamp
* signer identity or signer key reference
* signing purpose
* review status

This makes the record more useful for future verification.

## 5. Signature Scope

A signature may apply to different scopes.

### 5.1 Full Record Signature

A full record signature signs the entire canonical Q-Point record.

Use case:

* preserving the exact record state
* release snapshots
* archival records
* stable reviewed records

### 5.2 Source Binding Signature

A source binding signature signs the relationship between:

```text
Epicenter ID + source hash + content reference
```

Use case:

* proving that a Q-Point record refers to a specific source contribution
* binding a contribution to a trace record
* preventing silent source replacement

### 5.3 Reviewer Signature

A reviewer signature attests that a reviewer examined the record.

Use case:

* human review
* institutional review
* dispute review
* governance review

### 5.4 System Signature

A system signature attests that a record was generated by a specific tool, model, workflow, or platform.

Use case:

* GPT-generated records
* validator-generated records
* automated scoring pipeline output
* cross-AI aggregation output

### 5.5 Contributor Signature

A contributor signature may be used when a contributor wants to sign or acknowledge a contribution record.

Use case:

* creator acknowledgment
* voluntary source attestation
* contribution ownership claim for review purposes

Important:

A contributor signature is not automatically a legal ownership claim.

It is a source attestation signal.

## 6. Signature Types

The model defines the following recommended signature types.

| Signature Type             | Purpose                                                                    |
| -------------------------- | -------------------------------------------------------------------------- |
| `record_signature`         | Signs the canonical Q-Point record.                                        |
| `source_binding_signature` | Signs the binding between Epicenter ID, source hash, and source reference. |
| `reviewer_signature`       | Attests review by a human, organization, or hybrid reviewer.               |
| `system_signature`         | Attests generation by a tool, model, or workflow.                          |
| `contributor_signature`    | Attests contributor-side acknowledgment.                                   |
| `trace_binding_signature`  | Signs connection to Trace Protocol or related trace records.               |
| `release_signature`        | Signs a release snapshot or candidate version.                             |

## 7. Canonicalization

Before signing, the record should be canonicalized.

Canonicalization means transforming the record into a stable byte representation so that the same logical record produces the same hash.

Recommended canonicalization rules:

1. Use UTF-8 encoding.
2. Remove insignificant whitespace.
3. Sort object keys lexicographically.
4. Preserve array order.
5. Preserve numeric values as JSON-compatible numbers.
6. Exclude signature fields from the signed payload unless explicitly signing a signature envelope.
7. Convert YAML records to canonical JSON before hashing.
8. Use a clearly declared canonicalization method.

Recommended field:

```yaml
canonicalization:
  method: "json-canonicalization-scheme"
  version: "draft"
```

This model does not mandate a single canonicalization standard in v0.1.0.

Implementations should document the method used.

---

## 8. Hashing

A signed Q-Point record should include hashes.

Recommended hash algorithm:

```text
SHA-256
```

Possible hash fields:

| Field                    | Purpose                                                 |
| ------------------------ | ------------------------------------------------------- |
| `record_hash`            | Hash of the canonical Q-Point record.                   |
| `source_hash`            | Hash of the source contribution.                        |
| `schema_hash`            | Hash of the schema used for validation.                 |
| `scoring_model_hash`     | Hash of the scoring model document or version snapshot. |
| `trace_bundle_hash`      | Hash of related trace references.                       |
| `signature_payload_hash` | Hash of the exact payload signed.                       |

Example:

```yaml
hashes:
  record_hash:
    algorithm: "sha256"
    value: "sha256-record-placeholder"
  source_hash:
    algorithm: "sha256"
    value: "sha256-source-placeholder"
  schema_hash:
    algorithm: "sha256"
    value: "sha256-schema-placeholder"
  scoring_model_hash:
    algorithm: "sha256"
    value: "sha256-scoring-model-placeholder"
```

## 9. Signature Algorithms

This model recommends support for widely used digital signature approaches.

Possible algorithms:

| Algorithm    | Use Case                                        |
| ------------ | ----------------------------------------------- |
| `Ed25519`    | Recommended modern compact signature.           |
| `ECDSA-P256` | Common public-key signature option.             |
| `RSA-PSS`    | Legacy-compatible signature option.             |
| `JWS`        | JSON Web Signature envelope compatibility.      |
| `minisign`   | Lightweight file signing.                       |
| `GPG`        | Existing open-source release signing workflows. |

Recommended default for future implementations:

```text
Ed25519
```

However, this document does not require one mandatory algorithm at this stage.

The first priority is to define the record structure and binding logic.

A signed Q-Point record should bind at least the following fields:

```text

```

If a record is used for future allocation reference, it should also bind:

```text
trace_links
cross_ai
governance
privacy
anti_gaming
```

The exact signed field list should be declared.

Example:

```yaml
signature_scope:
  included_fields:
    - "record_id"
    - "protocol_version"
    - "created_at"
    - "epicenter"
    - "source"
    - "q_point_components"
    - "catalyst_vector"
    - "q_point_summary"
    - "status"
    - "disclaimer"
  excluded_fields:
    - "signatures"
```

## 11. Signature Envelope

A Q-Point record may include a `signatures` section.

Example structure:

```yaml
signatures:
  - signature_id: "sig-2026-000001"
    signature_type: "record_signature"
    signer:
      signer_id: "system-qpoint-evaluator-001"
      signer_type: "system"
      public_key_id: "key-2026-000001"
    signature_algorithm: "Ed25519"
    canonicalization:
      method: "json-canonicalization-scheme"
      version: "draft"
    signed_at: "2026-06-03T10:30:00+09:00"
    signature_scope:
      included_fields:
        - "record_id"
        - "protocol_version"
        - "created_at"
        - "epicenter"
        - "source"
        - "q_point_components"
        - "catalyst_vector"
        - "q_point_summary"
        - "status"
        - "disclaimer"
      excluded_fields:
        - "signatures"
    hashes:
      signature_payload_hash:
        algorithm: "sha256"
        value: "sha256-payload-placeholder"
      record_hash:
        algorithm: "sha256"
        value: "sha256-record-placeholder"
    signature_value: "base64url-signature-placeholder"
    verification_status: "unverified"
```

## 12. Verification Flow

A verifier should follow this process.

```text
1. Load the Q-Point record.
2. Check the declared protocol version.
3. Check the declared schema version.
4. Validate the record against the schema.
5. Remove or isolate signature fields according to the signature scope.
6. Canonicalize the signed payload.
7. Calculate the payload hash.
8. Compare calculated hash to declared hash.
9. Verify the signature using the declared public key.
10. Check timestamp, signer metadata, and verification status.
11. Check trace binding and Epicenter ID binding.
12. Return verification result.
```

Minimal verification result:

```yaml
verification_result:
  verified: true
  signature_id: "sig-2026-000001"
  record_hash_match: true
  signature_valid: true
  schema_valid: true
  signer_trusted: "unknown"
  verified_at: "2026-06-03T10:45:00+09:00"
```

---

## 13. Verification Status Values

Recommended verification status values:

| Value             | Meaning                                         |
| ----------------- | ----------------------------------------------- |
| `unverified`      | Signature is present but has not been verified. |
| `verified`        | Signature has been cryptographically verified.  |
| `failed`          | Signature verification failed.                  |
| `expired`         | Signature or key is expired.                    |
| `revoked`         | Signature or key has been revoked.              |
| `unknown_key`     | Public key could not be found.                  |
| `schema_mismatch` | Record does not match expected schema.          |
| `hash_mismatch`   | Hash does not match canonical payload.          |
| `disputed`        | Signature or record is under dispute.           |

---

## 14. Trust Levels

A valid signature does not automatically mean the content is true.

It only means the signed payload was signed by the holder of the corresponding key.

Recommended trust levels:

| Trust Level                | Meaning                                            |
| -------------------------- | -------------------------------------------------- |
| `cryptographically_valid`  | Signature is mathematically valid.                 |
| `source_bound`             | Signature is bound to source hash or Epicenter ID. |
| `schema_validated`         | Record passes schema validation.                   |
| `review_attested`          | Reviewer signature is present.                     |
| `institutionally_attested` | Trusted organization signature is present.         |
| `allocation_ready`         | Record may be considered for allocation review.    |
| `disputed`                 | Record requires dispute handling.                  |

The model should separate:

```text
Signature validity
Source attribution
Review quality
Allocation readiness
```

These are related but not identical.

## 15. Key Management

Each signer should be associated with a public key reference.

Example:

```yaml
signer:
  signer_id: "system-qpoint-evaluator-001"
  signer_type: "system"
  public_key_id: "key-2026-000001"
  public_key_reference: "https://example.org/keys/key-2026-000001.json"
```

Possible signer types:

```text
human
system
organization
reviewer
contributor
hybrid
unknown
```

Key management should support:

* public key discovery
* key rotation
* key expiration
* key revocation
* signer metadata
* signature timestamp
* optional trusted registry

This version does not define a mandatory key registry.

Future versions may define a `q-point-key-registry` format.

## 16. Timestamping

Every signature should include a timestamp.

Recommended field:

```yaml
signed_at: "2026-06-03T10:30:00+09:00"
```

Optional fields:

```yaml
timestamp_authority:
  type: "none"
  reference: null
```

Future versions may support:

* trusted timestamp authority
* blockchain timestamp anchoring
* transparency log timestamping
* Git commit timestamp binding
* release tag timestamp binding

At v0.1.0, ISO 8601 timestamps are sufficient for working draft use.

## 17. Relationship to Epicenter ID

The signature model should bind the Q-Point record to the Epicenter ID.

Example:

```yaml
epicenter_binding:
  epicenter_id: "qsrc-2026-000001"
  source_hash: "sha256-source-placeholder"
  binding_hash:
    algorithm: "sha256"
    value: "sha256-epicenter-binding-placeholder"
```

The Epicenter binding helps answer:

```text
Which contribution generated this Q-Point record?
```

This is essential for future trace review, gratitude, attribution, or allocation reference.

## 18. Relationship to Trace Protocol

Trace Protocol records evidence.

Q-Point records value traces.

The signature model binds them.

```text
Trace Protocol = Evidence layer
Q-Point Protocol = Value trace layer
Signature Model = Integrity and binding layer
```

A possible flow:

```text
Source contribution
  ↓
Epicenter ID
  ↓
Trace record
  ↓
Q-Point record
  ↓
Signature envelope
  ↓
Verification result
```

Recommended trace binding fields:

```yaml
trace_binding:
  trace_id: "trace-2026-000001"
  related_trace_ids:
    - "trace-2026-000002"
  trace_bundle_hash:
    algorithm: "sha256"
    value: "sha256-trace-bundle-placeholder"
```

## 19. Relationship to Royalty OS

Royalty OS may use reviewed Q-Point records as allocation reference data.

However, unsigned records may be weaker as evidence.

Signed records can improve:

* integrity
* source binding
* review readiness
* dispute handling
* auditability
* allocation review confidence

A possible future flow:

```text
Q-Point Record
  ↓
Signature Verification
  ↓
Trace Review
  ↓
Governance Review
  ↓
Allocation Reference
  ↓
Royalty OS
```

Important:

A valid signature does not automatically make a record royalty-ready.

Royalty readiness requires separate review, legal framework, consent, and governance.

## 20. Relationship to Royalty Point Maru

Royalty Point Maru may generate draft Q-Point evaluations and YAML records.

However, GPT-generated records should initially be treated as unsigned or system-draft records.

A GPT interface may generate:

```yaml
signature_status:
  signed: false
  signature_required: true
  generated_by: "Royalty Point Maru"
  review_required: true
```

If future Actions or external tools are added, Royalty Point Maru may send records to a signing service.

Recommended interpretation:

```text
Royalty Point Maru = Evaluation interface
Signature service = Verification and integrity layer
```

The GPT itself should not claim cryptographic signing unless connected to a real signing action or external service.

## 23. Privacy and Consent

Signatures can increase trust, but they can also increase privacy risk.

A signed record may permanently associate a contribution with a signer, timestamp, or key.

Implementations should consider:

* pseudonymous signatures
* private signatures
* selective disclosure
* contributor consent
* redaction workflows
* key rotation
* record suppression
* separation between identity and contribution

Avoid exposing personal identity unnecessarily.

For early-stage implementations, use signer IDs and public key references rather than real names when possible.

---

## 24. Anti-Gaming and Abuse Risks

Signature models can also be abused.

Possible abuse patterns:

* signing low-quality records to make them appear authoritative
* creating many pseudonymous signatures
* falsely claiming source authorship
* signing altered source summaries
* signing inflated Q-Point scores
* using signatures to imply legal entitlement
* using signatures as fake proof of payment rights

Recommended safeguards:

* schema validation
* source hash binding
* reviewer countersignature
* dispute status
* audit logs
* signature revocation
* transparent signer metadata
* non-monetary disclaimer
* separation between signature validity and allocation readiness

## 27. Summary

The Q-Point Record Signature Model defines a verification layer for Q-Point records.

It does not change Q-Point into money.

It does not create legal claims.

It does not create automatic royalty rights.

It makes Q-Point records more verifiable by binding:

* record content
* Epicenter ID
* source hash
* trace references
* schema version
* scoring model version
* timestamp
* signer metadata
* review context

Q-Point v0.2 records value traces.

The signature model prepares Q-Point v0.3 to verify those value traces.

In short:

```text
Q-Point v0.2:
  Measure the value trace.

Q-Point v0.3:
  Sign and verify the value trace.
```

This is the step from value measurement to verifiable value records.
