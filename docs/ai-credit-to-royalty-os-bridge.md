## 1. Purpose

This document defines a conceptual bridge between usage-side AI credits and future Royalty OS allocation systems.

The purpose of this bridge is to describe how value entering AI systems through usage fees, credits, subscriptions, or compute-based payments may be interpreted as a value pool, and how reviewed Q-Point records may serve as reference data for future attribution, gratitude, licensing, or royalty allocation.

This bridge does not automatically convert AI credits into money.

This bridge does not create a payment claim.

This bridge does not define a legally binding royalty mechanism.

It defines a possible reference model for future reviewed allocation.

In short:

```text
AI Credit Layer
  ↓
Q-Point Value Measurement Layer
  ↓
Royalty OS Allocation Reference Layer
```

---

## 2. Core Idea

AI-mediated systems increasingly receive value from users, developers, organizations, and platforms.

This value may appear as:

* AI credits
* usage-based fees
* API payments
* subscriptions
* compute credits
* inference credits
* platform-level AI usage pools

These are usage-side value inflows.

At the same time, AI systems are shaped by human questions, prompts, documents, concepts, protocols, and structural insights.

Q-Point Protocol records the non-monetary value trace of those contributions.

The AI Credit to Royalty OS Bridge connects these two directions:

```text
Human / Organization → AI
  = Usage-side value inflow

AI → Human / Contributor
  = Reviewed value return reference
```

The bridge describes how these two flows may become part of a future value circulation system.

## 3. Layer Model

The bridge has three layers.

### 3.1 AI Credit Layer

The AI Credit Layer represents value entering AI systems from usage.

Examples may include:

* credit-based AI usage
* inference-based billing
* subscription-based access
* usage-based API fees
* compute-based platform fees

In this document, this layer is called:

```text
Usage-Side Value Inflow
```

This is not treated as tax, legal revenue, or public money.

It is simply a value signal generated by AI usage.

---

### 3.2 Q-Point Layer

The Q-Point Layer records how human or hybrid contributions influence AI-mediated reasoning.

Q-Point may measure:

* originality
* depth
* catalyst
* trace persistence
* C-vector movement
* cross-AI resonance
* attribution confidence
* review status

This layer does not distribute value.

It measures contribution weight.

---

### 3.3 Royalty OS Layer

The Royalty OS Layer represents a future reviewed allocation system.

It may use Q-Point records as reference data for:

* attribution
* gratitude
* licensing review
* royalty weighting
* creator compensation
* institutional recognition
* value circulation

Royalty OS should not automatically execute payments from raw Q-Point scores.

Review, governance, consent, legal frameworks, and contractual rules are required.

## 5. Basic Conversion Model

Let:

| Symbol | Meaning                                                    |
| ------ | ---------------------------------------------------------- |
| `P`    | Value pool derived from AI usage-side inflow.              |
| `B`    | Allocation buffer rate.                                    |
| `Q_i`  | Q-Point score for contributor or contribution `i`.         |
| `Q_i*` | Adjusted Q-Point score for contribution `i`.               |
| `W_i`  | Normalized allocation weight for contribution `i`.         |
| `R_i`  | Allocation reference amount or ratio for contribution `i`. |

The minimal model is:

```text
W_i = Q_i / ΣQ
```

Then:

```text
R_i = P × B × W_i
```

Expanded:

```text
R_i = P × B × (Q_i / ΣQ)
```

This creates a simple proportional allocation model.

However, raw Q-Point scores should usually be adjusted before use.

---

## 6. Adjusted Q-Point Model

Raw Q-Point scores may not be sufficient for allocation reference.

A reviewed system should use adjusted Q-Point:

```text
Q_i* = Q_i × T_i × A_i × G_i
```

Where:

| Symbol | Meaning                        |
| ------ | ------------------------------ |
| `Q_i*` | Adjusted Q-Point score.        |
| `Q_i`  | Base Q-Point score.            |
| `T_i`  | Trace Persistence factor.      |
| `A_i`  | Attribution confidence factor. |
| `G_i`  | Governance review factor.      |

Then:

```text
W_i = Q_i* / ΣQ*
```

And:

```text
R_i = P × B × (Q_i* / ΣQ*)
```

This is the recommended bridge formula.

---

## 7. Recommended Bridge Formula

The recommended formula is:

```text
R_i = P × B × (Q_i* / ΣQ*)
```

Where:

```text
Q_i* = Q_i × T_i × A_i × G_i
```

This model ensures that allocation reference is based not only on raw score, but also on traceability, attribution reliability, and governance review.

## 8. Factor Definitions

### 8.1 Value Pool `P`

`P` represents the portion of usage-side value inflow considered for future allocation reference.

Examples may include:

* AI credit usage pool
* inference usage pool
* subscription-derived AI value pool
* API revenue pool
* institutional allocation pool

`P` must be defined by the implementing system.

This document does not require any platform to allocate or reserve such a pool.

---

### 8.2 Buffer Rate `B`

`B` is the allocation buffer rate.

It defines what portion of the value pool may be considered for reviewed allocation.

Example:

```text
B = 0.05
```

This means 5% of the relevant value pool may be considered for future allocation reference.

`B` may be set by:

* platform policy
* institutional governance
* licensing agreement
* experimental protocol
* public funding rule
* creator economy framework

---

### 8.3 Base Q-Point `Q_i`

`Q_i` is the base Q-Point score of contribution `i`.

It may be calculated using:

```text
Q = 0.30O + 0.25D + 0.30C + 0.15T
```

Where:

* `O` = Originality
* `D` = Depth
* `C` = Catalyst
* `T` = Trace Persistence

The Catalyst score may use the C-vector:

```text
C = 0.35S + 0.25Δ + 0.25K + 0.15V
```

Where:

* `S` = Self-Question Trigger
* `Δ` = Confidence Shift
* `K` = Internal Conflict Detection
* `V` = Viewpoint Shift

---

### 8.4 Trace Persistence Factor `T_i`

`T_i` measures how reliably a contribution can be traced across time, records, systems, or references.

Possible signals include:

* stable Epicenter ID
* trace links
* content hash
* source metadata
* citation records
* cross-AI references
* persistent reuse records

`T_i` should be normalized:

```text
0.0 <= T_i <= 1.0
```

---

### 8.5 Attribution Confidence Factor `A_i`

`A_i` measures how confidently a contribution can be attributed to a source.

Possible signals include:

* clear source record
* low ambiguity
* timestamp reliability
* hash verification
* human review
* low dispute risk
* strong trace evidence

`A_i` should be normalized:

```text
0.0 <= A_i <= 1.0
```

---

### 8.6 Governance Review Factor `G_i`

`G_i` measures whether the contribution has passed appropriate review.

Possible values may include:

| Status                   | Suggested `G_i` |
| ------------------------ | --------------- |
| Unreviewed               | `0.0` to `0.3`  |
| Draft review             | `0.3` to `0.6`  |
| Human reviewed           | `0.6` to `0.9`  |
| Institutionally approved | `0.9` to `1.0`  |
| Disputed                 | `0.0` or paused |

`G_i` should not be used as a truth score.

It represents governance readiness.

## 9. Example Calculation

Assume:

```text
P = 10000
B = 0.05
```

The allocation reference pool is:

```text
P × B = 10000 × 0.05 = 500
```

Assume three reviewed contributions:

| Contribution | Base Q | Trace `T_i` | Attribution `A_i` | Governance `G_i` |
| ------------ | -----: | ----------: | ----------------: | ---------------: |
| A            |   0.84 |        0.90 |              0.85 |             0.80 |
| B            |   0.72 |        0.70 |              0.80 |             0.60 |
| C            |   0.65 |        0.95 |              0.90 |             0.90 |

Adjusted Q-Point:

```text
Q_A* = 0.84 × 0.90 × 0.85 × 0.80 = 0.51408
Q_B* = 0.72 × 0.70 × 0.80 × 0.60 = 0.24192
Q_C* = 0.65 × 0.95 × 0.90 × 0.90 = 0.500175
```

Total:

```text
ΣQ* = 1.256175
```

Weights:

```text
W_A = 0.51408 / 1.256175
W_B = 0.24192 / 1.256175
W_C = 0.500175 / 1.256175
```

Allocation reference:

```text
R_A = 500 × W_A
R_B = 500 × W_B
R_C = 500 × W_C
```

This produces a reviewed proportional allocation reference.

It is not an automatic payment.

## 11. Required Safeguards

Any implementation of this bridge should include safeguards.

### 11.1 Non-Automatic Conversion

AI credits should not be automatically converted into contributor payments by this bridge alone.

The bridge only defines reference weights.

### 11.2 Human or Institutional Review

Allocation should require review when:

* value is monetary
* contributors are disputed
* records have low attribution confidence
* records involve sensitive data
* multiple contributors claim the same source
* platform policy is unclear

### 11.3 Legal and Contractual Layer

Any real distribution requires a separate legal and contractual framework.

This may include:

* licensing terms
* contributor consent
* platform agreement
* tax handling
* payout compliance
* jurisdictional rules
* dispute resolution

### 11.4 Privacy Protection

The bridge should not expose personal identity unnecessarily.

Use contribution IDs, Epicenter IDs, hashes, or pseudonymous identifiers where possible.

### 11.5 Anti-Gaming

The bridge should not reward artificial score inflation.

Safeguards may include:

* duplicate detection
* novelty checks
* delayed influence scoring
* human review
* confidence scoring
* trace verification
* dispute handling

## 16. Governance Model

A governance model should define:

| Role                  | Responsibility                                       |
| --------------------- | ---------------------------------------------------- |
| `pool_operator`       | Defines the value pool and buffer rate.              |
| `q_point_evaluator`   | Provides Q-Point records and scores.                 |
| `trace_reviewer`      | Checks trace persistence and attribution.            |
| `governance_reviewer` | Reviews eligibility and disputes.                    |
| `royalty_operator`    | Executes any future allocation under separate rules. |
| `contributor`         | Provides or is associated with the contribution.     |

No single automated score should be treated as final for monetary allocation.

## 17. Risk Notes

### 17.1 Financial Misinterpretation

Users may misunderstand bridge reference units as guaranteed money.

Mitigation:

* include disclaimers
* avoid “earned” language
* use “allocation reference” instead of “payment”
* require review before allocation

### 17.2 Platform Capture

A platform may use allocation logic to centralize power.

Mitigation:

* open schemas
* exportable records
* transparent formulas
* independent review
* interoperable IDs

### 17.3 Low-Quality Score Optimization

Users may attempt to produce prompts optimized for high Q-Point scores.

Mitigation:

* anti-gaming checks
* delayed scoring
* human review
* duplicate detection
* trace verification

### 17.4 Attribution Disputes

Multiple contributors may claim the same contribution lineage.

Mitigation:

* trace records
* timestamps
* hashes
* review process
* dispute state
* partial allocation handling
